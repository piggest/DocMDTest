# プロダクト開発の哲学

プロダクト開発とは、つまるところ人間の問題を解くことである。技術は手段に過ぎない。にもかかわらず、私たちはしばしば技術そのものに魅了され、本来の目的を見失う。

---

## なぜ私たちは作るのか

### 問題の発見

よいプロダクトは、よい問いから生まれる。「何を作るか」より先に「誰のどんな困りごとを解くのか」を問うべきだ。

ユーザーインタビューを重ねるうち、あることに気づいた。人々が求めているのは機能ではなく、**結果**だということだ。旅行アプリに求めているのは「フライト検索」ではなく「家族と過ごす夏休み」なのである。

### 余白の設計

プロダクトの価値は、しばしば「ないもの」によって決まる。シンプルさは怠惰の産物ではなく、意図的な選択の結晶だ。

> 完璧とは、これ以上付け加えるものがなくなったときではなく、これ以上取り去るものがなくなったときに達成される。  
> — アントワーヌ・ド・サン＝テグジュペリ

## 開発プロセスの実際

### イテレーションの哲学

私たちのチームは3つの原則を掲げている。

- **早く失敗し、早く学ぶ** — 仮説は最小限の実験で検証する
- **ユーザーとともに作る** — 傍観者ではなく、共同制作者として
- **完璧より完成を選ぶ** — 届けないプロダクトに価値はない

優先順位の付け方も変わった。かつては機能の多さを誇っていたが、今は削った機能の数を誇る。

### 計測と直感のあいだ

| アプローチ | 強み | 弱点 |
|-----------|------|------|
| データドリブン | 再現性が高い | 未知の問いを立てられない |
| 直感ドリブン | 革新的な発見 | 再現・説明が難しい |
| 仮説検証型 | 両者のバランス | 時間がかかる |
